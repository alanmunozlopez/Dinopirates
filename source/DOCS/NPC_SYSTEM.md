# NPC System

NPCs are static, interactive entities (computers, terminal cats, phones) that display conditional dialogs and can deliver items or keys to the player one time only. Dialog is always repeatable; the grant is one-shot.

> See also: [TRIGGER_SYSTEM.md](TRIGGER_SYSTEM.md) for the Trigger system (parallel to NPCs). [DIALOG_SYSTEM.md](DIALOG_SYSTEM.md) for how `addScreen` works. [SCRIPTS_TRIGGERS_NPC_SCHEMA.md](SCRIPTS_TRIGGERS_NPC_SCHEMA.md) for the full customFields schema.

---

## 1. NPC Class — Constructor and All Fields

File: `entities/props/npc.lua`.

```lua
function NPC:init(x, y, npcType, iid, room, sourceFeed)
```

| Parameter | Type | Description |
|-----------|------|-------------|
| `x`, `y` | Number | Position in the room (LDtk coordinates, sprite center). |
| `npcType` | String | Name of the animation state in the spritesheet (`"cat"`, `"computer"`). |
| `iid` | String | Unique UUID generated by LDtk. Used to look up the entity in `levelsLDTK` and persist `hasGranted`. |
| `room` | Number | Index in `levelsLDTK` for the current room. |
| `sourceFeed` | Number | Index of the dialog video feed. Passed to `dialogUI:addScreen(script, sourceFeed)`. Default: `0`. |

### Instance Fields

| Field | Initial value | Description |
|-------|---------------|-------------|
| `self.npcType` | npcType | Active animation state. |
| `self.iid` | iid | UUID for lookup in levelsLDTK. |
| `self.room` | room | Index in levelsLDTK. |
| `self.sourceFeed` | sourceFeed or 0 | Portrait feed for the dialog. |
| `self.script` | nil | Required by MazeScene (`grantAchievementIfNeeded(trigger.script)`); nil is valid. |
| `self.type` | nil | Required by `state.lua` to check `currentTrigger.type`; nil activates the generic "Press A" HUD icon. |
| `self.wall` | NPCCollider | Solid collider sprite created in the constructor. |

### Sprite Configuration

```lua
NPC.super.init(self, 'assets/images/props/npc', true)  -- loads spritesheet
self.animation:addState('cat',      1, 4)   -- frames 1-4 for the cat
self.animation.cat.frameDuration = 12
self.animation:addState('computer', 5, 8)   -- frames 5-8 for the computer
self.animation:setState(npcType)

self:setSize(32, 32)
self:setCollideRect(0, 0, 32, 32)
self:setZIndex(ZIndex.props)              -- ZIndex = 2
self:setGroups(CollideGroups.props)       -- group 3
self:setCollidesWithGroups({})            -- the NPC detects nothing
self:add(x, y)
```

### Base Class

`NPC` extends `NobleSprite` (not `Graphics.sprite`). This gives it access to the `self.animation` system.

## 2. NPCCollider — Why It Exists and Collision Groups

Each NPC instantiates an additional `NPCCollider`:

```lua
self.wall = NPCCollider(x, y)
```

`NPCCollider` extends `NobleSprite` and is an **invisible** `24×24` px sprite.

```lua
function NPCCollider:init(x, y)
    self:setSize(24, 24)
    self:setCollideRect(0, 0, 24, 24)
    self:setGroups(CollideGroups.wall)   -- group 5 (walls)
    self:setCollidesWithGroups({})       -- detects nothing
    self:add(x, y)
end
```

**Why it exists**: The visual `NPC` sprite belongs to the `props` group (3) and returns `'overlap'` on collision — the player physically passes through it to trigger the interaction. However, the NPC must block the player's movement so they don't visually walk through it. `NPCCollider` is in the `wall` group (5), which the player treats as a solid surface (`return 'freeze'`).

| Sprite | Class | Size | Group | Collision Response |
|--------|-------|------|-------|--------------------|
| Visual | `NPC` | 32×32 | `CollideGroups.props` (3) | `'overlap'` → sets `currentTrigger` |
| Physics blocker | `NPCCollider` | 24×24 | `CollideGroups.wall` (5) | `'freeze'` → blocks movement |

The `NPCCollider` is smaller than the visual sprite to allow the player to approach without being blocked from a distance.

### Cleanup on Remove

```lua
function NPC:remove()
    if self.wall then
        Noble.currentScene():removeSprite(self.wall)
        self.wall = nil
    end
    NPC.super.remove(self)
end
```

Both sprites are removed together. Noble Engine does not manage the `NPCCollider` automatically because it is a separate sprite.

## 3. Conditional Script System — Complete Format

The main interaction is `NPC:returnScript()`, called from `MazeScene.AButtonDown` when `player.currentTrigger` is an NPC.

### `returnScript()` Flow

```
returnScript()
  1. scriptName, grantsStr = self:evaluateConditions()
  2. If grantsStr exists and not self:hasGranted():
       self:applyGrant(grantsStr)
       self:markGranted()
  3. Returns scriptName or "" if there was no match
```

### `evaluateConditions()` — Top-to-Bottom Evaluation

Reads `cf.conditionalScripts` from the entity in `levelsLDTK`. Evaluates each entry in order:

```
For each entry in conditionalScripts:
  Split into parts by ':' using gmatch("[^:]+")
  parts[1] = conditionExpr
  parts[2] = scriptName
  parts[3] and parts[4] = grantKey and grantVal (optional)

  If conditionExpr and scriptName and evaluateCondition(conditionExpr):
    Returns (scriptName, grantKey..":"..grantVal)

If no entry applies: returns (nil, nil)
```

### Entry Format in `conditionalScripts`

```lua
-- Condition and script only (no grant)
"condition:scriptName"

-- Condition, script, and grant (four parts separated by ':')
"condition:scriptName:grantKey:grantVal"
```

### `evaluateCondition(conditionExpr)` — Supported Operators

| Form | Example | Evaluation |
|------|---------|-----------|
| Catch-all | `true` | Always `true` (special case, does not query PlayerData) |
| Numeric comparison | `battery>20` | `PlayerData.battery > 20` |
| Numeric comparison | `mapPercent>=50` | `PlayerData.mapPercent >= 50` |
| Direct boolean | `items.hasLamp` | `PlayerData.items.hasLamp == true` |
| Negated boolean | `!isTiny` | `PlayerData.isTiny ~= true` |

Supported numeric operators: `>`, `<`, `>=`, `<=`, `==`, `!=`

The path is traversed using `gmatch("[^%.]+")` on `PlayerData`. If the field does not exist or is not a number in numeric comparisons, `0` is assumed.

### Example `conditionalScripts` in LDtk

```lua
-- Cat NPC that evaluates the player's inventory
conditionalScripts = {
    "!items.hasLamp:catNoLamp",      -- if they don't have the lamp
    "!items.hasBoots:catNoBoots",    -- if they have the lamp but not the boots
    "true:catWhat"                   -- fallback: always applies
}
```

## 4. Grant System — Formats, Persistence, and `hasGranted`

### Grant Format

Grants are specified as parts 3 and 4 of a `conditionalScripts` entry:

```
"condition:script:grantKey:grantVal"
```

### `applyGrant(grantsStr)` — What Can Be Granted

```lua
function NPC:applyGrant(grantsStr)
    local grantKey, grantVal = grantsStr:match("^([^:]+):(.+)$")
    grantKey = grantKey:gsub("%s+", "")  -- removes spaces

    if grantKey == "key" then
        local keyNum = tonumber(grantVal)
        PlayerData.keys[keyNum] = true       -- grants key N
    elseif grantVal == "true" then
        PlayerData.items[grantKey] = true    -- grants boolean item
    end
end
```

| Grant format | Example | Effect on PlayerData |
|-------------|---------|---------------------|
| `key:N` | `key:2` | `PlayerData.keys[2] = true` |
| `fieldName:true` | `hasBoots:true` | `PlayerData.items.hasBoots = true` |

An NPC can give **a single grant per `conditionalScripts` entry**. For multiple grants, use separate entries with appropriate conditions, or resolve the second grant through script logic.

### `hasGranted` — One-Shot Flag

- `hasGranted` lives in `levelsLDTK[room].entities.NPC[i].customFields.hasGranted`.
- Checked with `NPC:hasGranted()` before applying the grant.
- Marked with `NPC:markGranted()` after applying the grant.
- **Dialog remains repeatable** even when `hasGranted == true`. Only the grant is one-shot.
- The SaveSystem persists `hasGranted` automatically when exiting the room.

### Complete Grant Flow

```
Player presses A → MazeScene.AButtonDown
  scriptName, grantsStr = npc:returnScript()
    ↓
  evaluateConditions() returns ("terminalAuth", "key:3")
    ↓
  hasGranted() → false (first time)
    ↓
  applyGrant("key:3") → PlayerData.keys[3] = true
    ↓
  markGranted() → levelsLDTK[room].NPC[i].customFields.hasGranted = true
    ↓
  dialogUI:addScreen("terminalAuth", sourceFeed)

Second interaction:
  evaluateConditions() returns ("terminalAuth", "key:3")
  hasGranted() → true → applyGrant is skipped
  dialogUI:addScreen("terminalAuth", sourceFeed)  ← dialog still shows
```

## 5. `sourceFeed` — What It Does

`sourceFeed` is an integer passed as the second argument to `dialogUI:addScreen(scriptName, sourceFeed)`. It indicates which portrait feed to use in the dialog's videoFeed.

In the current `dialogScreen` code, the second argument exists in the signature but the internal system uses the `video` field of each `DialogLine` in `script.lua` to determine the portrait. `sourceFeed` is available for future extensions or for specific NPC video feeds that may differ from the standard player portrait.

Default value in LDtk: `0`. If not specified in the customField, MazeScene uses `cf.sourceFeed or 0`.

## 6. Update Loop — Dynamic Z-Index

```lua
function NPC:update()
    self:setZIndex(self.y)
end
```

`NPC:update()` is called every frame. It adjusts the ZIndex to the current value of `self.y` (the sprite's Y coordinate in pixels). This implements y-sorting: NPCs that are lower on the screen (higher Y) render on top of those that are higher up. Same pattern as `PropItem`.

`NPCCollider` has no `update()` — its ZIndex is fixed and irrelevant (invisible sprite).

## 7. Spawning in MazeScene

Spawn block in `MazeScene:enter()`:

```lua
local npcEntities = levelsLDTK[room].entities
if npcEntities and npcEntities.NPC then
    for _, npcData in ipairs(npcEntities.NPC) do
        local cf = npcData.customFields or {}
        NPC(npcData.x, npcData.y,
            cf.type or "computer",
            npcData.iid,
            room,
            cf.sourceFeed or 0)
    end
end
```

**There is no skip condition for NPCs**: unlike Triggers (which are skipped if `usedTrigger == true`), NPCs are always instantiated. The `hasGranted` flag only affects whether the grant is applied, not whether the NPC appears.

### Collision That Sets `currentTrigger`

```lua
-- player/collisions.lua
elseif other:isa(NPC) then
    self.currentTrigger = other
    return 'overlap'
```

The player sets `currentTrigger` when overlapping with the NPC. MazeScene shows the "Press A" HUD icon. On pressing A:

```lua
-- MazeScene.AButtonDown
elseif player.currentTrigger and PlayerData.isGaming == true then
    local trigger = player.currentTrigger
    PlayerData.isGaming = false
    PlayerData.isTalking = true
    player.dialogUI:addScreen(trigger:returnScript(), trigger.sourceFeed)
    Utilities.grantAchievementIfNeeded(trigger.script)  -- trigger.script is nil for NPC
end
```

## 8. Persistence

| Field | Entity | When it changes | Who changes it |
|-------|---------|-----------------|----------------|
| `hasGranted` | NPC (in levelsLDTK) | First interaction with a successful grant | `NPC:markGranted()` |

`NPC:getLDTKData()` finds the entity by `iid` in `levelsLDTK[self.room].entities.NPC`. If the room or entity does not exist, it returns `nil` safely.

The SaveSystem serializes `hasGranted` along with other modified fields when exiting the room. No extra steps are needed.

## Love2D Notes

### Overlap Detection

In Playdate, `collisionResponse` returns `'overlap'`. In Love2D with bump.lua:

```lua
-- Player collision filter
if other.type == "npc" then return "cross" end

-- In the update loop, detect overlap
local cols = world:queryRect(player.x, player.y, player.w, player.h)
for _, item in ipairs(cols) do
    if item.type == "npc" then
        currentTrigger = item.npcRef
        showHudIcon("pressA")
    end
end
```

### Two Sprites in Love2D

Create two separate colliders in bump.lua: one with type `"npc"` (overlap, activates interaction) and another with type `"wall"` (solid, blocks movement):

```lua
world:add({type="npc", npcRef=npc}, x, y, 32, 32)
world:add({type="wall"}, x+4, y+4, 24, 24)  -- centered solid blocker
```

### Y-Sort

In Love2D, sort the sprite list by `y` before drawing, or use a library like FLUX with `z` sorting.

### `hasGranted` Persistence

Save using the same save system as `usedTrigger`. Serialize by `iid`.
