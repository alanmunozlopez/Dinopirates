# Trigger System

Triggers are invisible entities placed in LDtk that activate dialogs, cutscenes, or other effects when the player enters contact with them or presses A. This document covers all their fields, types, conditional evaluation logic, and persistence.

> See also: [DIALOG_SYSTEM.md](DIALOG_SYSTEM.md) for the dialog engine. [CUTSCENE_SYSTEM.md](CUTSCENE_SYSTEM.md) for cutscene flow. [SCRIPTS_TRIGGERS_NPC_SCHEMA.md](SCRIPTS_TRIGGERS_NPC_SCHEMA.md) for the full customFields schema.

---

## 1. Trigger Entity — Fields

The constructor in `entities/props/trigger.lua` receives:

```lua
function Trigger:init(x, y, width, height, script, iid, room, type)
```

| Field | Type | Description |
|-------|------|-------------|
| `x`, `y` | Number | Center of the rect in LDtk coordinates. The sprite is positioned at `(x - width/2, y - height/2)`. |
| `width`, `height` | Number | Dimensions of the collision area. |
| `script` | String | Name of the script in `script.lua` executed as a fallback when `conditionalScripts` is empty or no condition applies. |
| `iid` | String | Unique UUID generated by LDtk. Used to look up the entity in `levelsLDTK` and for the SaveSystem to identify it when persisting `usedTrigger`. |
| `room` | Number | Index (position) within the `levelsLDTK` table. **Not** the room number; it is the index returned by `setFloor()`. |
| `type` | String | Trigger type. Controls how it activates and what HUD icon it displays. See section 2. |

In LDtk, in addition to the above fields, each Trigger has the following `customFields`:

| customField | Type | Description |
|-------------|------|-------------|
| `script` | String | Fallback script. Same value as the `script` field in the constructor. |
| `type` | String | Trigger type (`"Story"`, `"Search"`, `"Call"`, `"Cutscene"`, `"Counter"`, or `nil`). |
| `conditionalScripts` | Array\<String\> | List of conditions evaluated top-to-bottom before using the fallback `script`. |
| `usedTrigger` | Bool | `false` by default. When set to `true`, the trigger is not instantiated on the next room load. Persisted by the SaveSystem. |
| `mapPercent` | Number | Legacy field present in the data. Not currently used in condition evaluation. |

**Base class**: `Trigger` extends `Graphics.sprite` directly (the Playdate SDK base class), **not** `NobleSprite`.

**Collision groups**: The trigger belongs to group `3` (props). The player has `props` in its `collidesWithGroups`, so collision is detected normally.

**ZIndex**: `3` (fixed, does not change).

## 2. Trigger Types — Differences

The `type` field determines whether the trigger activates automatically (by collision) or manually (player presses A), and which HUD icon is shown.

### Automatic types (activated by collision)

| type | Behavior |
|------|----------|
| `"Story"` | On collision, calls `dialogUI:addScreen(returnScript(), sourceFeed)` directly. Shows dialog without player interaction. In the legacy fallback, it is **consumed** (`usedTrigger = true`). |
| `"Cutscene"` | On collision, sets `PlayerData.isGaming = false` and `PlayerData.isCutscene = true`. Calls `returnScript()` (which marks `usedTrigger = true`) and removes the sprite. The Panels cutscene is started from `MazeScene:enter()` when `comic_name` is detected in the room, not from the trigger. |
| `"Counter"` | On collision, increments `PlayerData.storyCounter` and removes itself immediately. Does not call `returnScript()`. Always consumed. |

### Manual types (require pressing A)

| type | HUD Icon | Behavior |
|------|----------|----------|
| `"Search"` | Investigate / magnifier | Player must press A. This is the **only type** that, in the legacy fallback (without `conditionalScripts`), is **not** automatically consumed — the trigger remains active. |
| `"Call"` | Radio / ring | Player must press A. Consumed in legacy fallback. |
| `nil` (no type) | Generic Press A | Player must press A. Consumed in legacy fallback. |

When the player enters collision with a manual trigger, `Player:collisionResponse()` assigns `self.currentTrigger = other`. `MazeScene.AButtonDown` detects `player.currentTrigger` and executes:

```lua
PlayerData.isGaming = false
PlayerData.isTalking = true
player.dialogUI:addScreen(trigger:returnScript(), trigger.sourceFeed)
Utilities.grantAchievementIfNeeded(trigger.script)
```

## 3. `returnScript()` — Complete Logic

`returnScript()` is the central method. It is called when the trigger activates to determine which script to run and whether the trigger should be marked as used.

```
Trigger:returnScript()
  1. Clears the sprite's collideRect (clearCollideRect)
  2. Finds roomData = levelsLDTK[self.room]
  3. If no data → returns self.script (fallback)
  4. Finds triggerData in roomData.entities.Triggers by iid
  5. If not found → returns self.script
  6. Reads cf = triggerData.customFields
  7. If cf.conditionalScripts exists and has entries:
     For each condStr in the list (top-to-bottom):
       a. Extracts conditionExpr and targetScript by splitting on ':'
       b. Detects terminal flag '!' at end of targetScript
       c. If '!' is present: isTerminal=true, removes '!' from name
       d. Evaluates conditionExpr (see section 4)
       e. If condition is met:
            - If isTerminal: cf.usedTrigger = true
            - Returns targetScript
  8. If no condition applies (legacy fallback):
     - If type != "Search": cf.usedTrigger = true
     - Returns self.script
```

The method modifies `cf.usedTrigger` directly in the live `levelsLDTK` table. The SaveSystem persists it to disk when exiting the room.

## 4. Condition Format — All Supported Operators

Conditions are written as strings in `conditionalScripts`. Each entry has the format:

```
"condition:scriptName"
"condition:scriptName!"
```

The condition is evaluated against `PlayerData`. If met, `scriptName` is returned.

### 4a. Numeric Comparisons

Format: `dotPath OPERATOR value`

```
"battery>20:dialogWithBattery"
"mapPercent>=50:midGameDialog"
"healthPoints<3:lowHPWarning"
"storyCounter==5:storyMoment"
"storyCounter!=0:notFirst"
```

Supported operators: `>`, `<`, `>=`, `<=`, `==`, `!=`

The path is traversed using `gmatch("[^%.]+")` on `PlayerData`. If the value does not exist or is not a number, `0` is assumed.

**Evaluation example**:
- Condition: `"battery>20"`
- Path: `"battery"` → `PlayerData.battery`
- Operator: `>`, Value: `20`
- Result: `PlayerData.battery > 20`

### 4b. Boolean Conditions (PlayerData paths)

Format: `dotPath` (true if the field is `true`)

```
"isTiny:scriptIfSmall"
"items.hasLamp:scriptWithLamp"
"items.hasBoots:scriptWithBoots"
"keys.1:scriptWithKey1"
```

The path is traversed the same way as numeric comparisons. The result is `(value == true)`.

### 4c. Boolean Negation

Prefix `!` before the path:

```
"!isTiny:scriptIfNormal"
"!items.hasLamp:scriptWithoutLamp"
```

Equivalent to `(value ~= true)`.

### 4d. Catch-all `"true"`

The literal `"true"` always matches. Used as the last fallback in the list:

```lua
conditionalScripts = {
    "isTiny:hugeXmas",
    "true:whyXmas"   -- always applies if none of the above matched
}
```

**Note**: `"true"` is not a PlayerData path; it is a special case recognized before normal parsing only in the NPC evaluator. In a Trigger, `"true"` is evaluated as a boolean path (does not exist in PlayerData → result is `false`). To guarantee a fallback in Triggers, use the `script` field in LDtk rather than `"true"` in `conditionalScripts`.

## 5. Terminal Flag `!` — When the Trigger is Marked as Used

The `!` suffix at the end of a script name in a `conditionalScripts` entry indicates the trigger should be consumed when that condition is met.

**Example**:
```lua
conditionalScripts = {
    "!isTiny:lockedMessage",     -- no !, trigger can be reused
    "isTiny:secretPassed!"       -- with !, marks usedTrigger=true when met
}
```

- Without `!`: the condition applies, the script runs, the trigger **remains active** (repeatable).
- With `!`: the condition applies, the script runs, `cf.usedTrigger = true` in `levelsLDTK` → the trigger will not be instantiated the next time the room is loaded.

**Legacy fallback** (when `conditionalScripts` is empty or no condition applies):
- If `type == "Search"`: the trigger is **not** consumed.
- For any other type: the trigger **is** consumed (`usedTrigger = true`).

## 6. `grantKey:grantVal` Format — What Can Be Granted

Triggers do not have their own grant system. Grants (keys, items) are the responsibility of NPCs. If a trigger needs to grant something, the solution is to use a dialog script that describes the grant and handle the grant logic in code (for example, through `Items` entities in the same room or story events).

For one-shot interaction grants, use an NPC with `conditionalScripts` that includes `grantKey:grantVal` (see [NPC_SYSTEM.md](NPC_SYSTEM.md)).

## 7. Persistence in `levelsLDTK` (`usedTrigger`)

The `usedTrigger` field lives in `levelsLDTK[room].entities.Triggers[i].customFields.usedTrigger`.

Persistence flow:

```
returnScript() modifies cf.usedTrigger = true in the live levelsLDTK table
        │
        ▼
MazeScene:finish() or pause() call SaveSystem.save()
        │
        ▼
SaveSystem serializes modified fields by iid to the datastore
        │
        ▼
On the next game load, SaveSystem.load() restores usedTrigger=true
        │
        ▼
MazeScene:enter() — Triggers block:
  if not used then Trigger(...) end
  -- If usedTrigger=true, the trigger is not instantiated
```

The trigger disappears permanently once marked. There is no way to reset it except by deleting the save or restarting from the original backup (`SaveSystem.createOriginalBackup()`).

## 8. Player Detection — Collision and Optimization

### Collision Detection

The Trigger uses an invisible `collideRect` of the same size as `width × height`. The player has `collideGroups = {props}` in its collision group list, so collision is detected via the Playdate SDK sprite engine.

`Player:collisionResponse(other)` classifies the collision:

```lua
elseif other:isa(Trigger) then
    if other.type == "Cutscene" then
        -- activates cutscene automatically
    elseif other.type == "Search" then
        self.currentTrigger = other
    elseif other.type == "Call" then
        self.currentTrigger = other
    elseif other.type == "Story" then
        PlayerData.isGaming = false
        self.dialogUI:addScreen(other:returnScript(), other.sourceFeed)
    elseif other.type == nil then
        self.currentTrigger = other
    elseif other.type == "Counter" then
        PlayerData.storyCounter += 1
        other:remove()
    end
    return 'overlap'
```

The `'overlap'` return allows the player to pass through the trigger without being blocked.

### Distance Optimization

`Config.Player.triggerCheckDist = 5` defines how many pixels the player must move before re-checking triggers. This value is consulted in the player movement system to avoid collision queries every frame.

### `currentTrigger`

For manual triggers (`Search`, `Call`, `nil`, `NPC`), the player stores the reference in `player.currentTrigger`. MazeScene displays the corresponding HUD icon while `currentTrigger` is assigned, and clears it when the player leaves the area.

## 9. Spawn Conditions in MazeScene

Spawn block in `MazeScene:enter()`:

```lua
if entities and entities.Triggers then
    for i, triggerData in ipairs(entities.Triggers) do
        local cf = triggerData.customFields or {}
        local used = cf.usedTrigger or false

        if not used then
            Trigger(triggerData.x, triggerData.y,
                    triggerData.width, triggerData.height,
                    cf.script, triggerData.iid, room, cf.type)
        end
    end
end
```

Only triggers with `usedTrigger == false` are instantiated. Already-used triggers simply do not exist in the scene.

## Love2D Notes

### Trigger as bump.lua Sensor

Replace `Graphics.sprite` with a ghost object in bump.lua:

```lua
world:add({type="trigger", triggerType="Search", script="myScript", iid=iid},
    x - w/2, y - h/2, w, h)

-- In the player's collision filter:
if item.type == "trigger" then return "cross" end
```

### Condition Evaluation

`returnScript()` and `conditionalScripts` are pure Lua logic (string parsing + querying fields from `PlayerData`). The evaluator can be copied directly — it has no Playdate SDK dependencies.

### Manual Trigger with HUD

```lua
-- In love.update: detect overlap
if overlaps(player, trigger) and not playerData.isTalking then
    currentTrigger = trigger
    showHudIcon(trigger.triggerType)  -- "Search", "Call", etc.
end

-- In love.keypressed:
if key == "return" and currentTrigger then
    playerData.isGaming = false
    playerData.isTalking = true
    dialogUI:addScreen(currentTrigger:returnScript(), currentTrigger.sourceFeed)
end
```

### Cutscene Flag

```lua
if trigger.triggerType == "Cutscene" then
    playerData.isGaming = false
    CutsceneManager:play(trigger:returnScript())
    trigger:remove()
end
```
